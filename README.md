# AES ECDH Tool

GUI-приложение для обмена зашифрованными сообщениями.

Используется схема:

```text
X25519 ECDH -> HKDF-SHA256 -> AES-256-GCM
```

Программа позволяет двум пользователям выработать общий секрет, проверить его через код проверки и шифровать сообщения через AES-256-GCM.

---

## Возможности

- генерация временной пары ключей X25519;
- обмен публичными ключами;
- выработка общего секрета через X25519 ECDH;
- получение AES-ключей через HKDF-SHA256;
- отдельный ключ для направления Alice -> Bob;
- отдельный ключ для направления Bob -> Alice;
- шифрование сообщений через AES-256-GCM;
- проверка целостности сообщений;
- код проверки для защиты от MITM-атаки;
- GUI на Tkinter;
- сборка в `.exe` через PyInstaller.

---

## Важно

Код проверки не является ключом шифрования.

Он нужен только для того, чтобы Alice и Bob убедились, что получили один и тот же общий секрет и что публичные ключи не были подменены.

Код проверки можно произносить вслух или сверять по другому доверенному каналу.

Если код проверки не совпадает, использовать соединение нельзя.

---

## Используемая криптография

В программе используются стандартные криптографические примитивы из библиотеки `cryptography`:

- `X25519` — выработка общего секрета;
- `HKDF-SHA256` — получение ключей из общего секрета;
- `AES-256-GCM` — шифрование и проверка целостности;
- `os.urandom` — генерация nonce.

Криптографические алгоритмы не реализуются вручную.

---

## Общая схема работы

```text
Alice private key + Bob public key
                ↓
             X25519
                ↓
        shared secret
                ↓
          HKDF-SHA256
                ↓
 ┌────────────────────────────┐
 │ AES key Alice -> Bob       │
 │ AES key Bob -> Alice       │
 │ verification code          │
 └────────────────────────────┘
```

## Защита от MITM

Без проверки кода возможна атака посредника:

```text
Alice <-> Attacker <-> Bob
```

В этом случае злоумышленник может подменить публичные ключи, выработать отдельный ключ с Alice и отдельный ключ с Bob, читать сообщения и пересылать их дальше.

Чтобы этого избежать, Alice и Bob должны сравнить код проверки по доверенному каналу.

Подходящие варианты:

- лично;
- голосом;
- по видеозвонку;
- через уже доверенный мессенджер;
- через другой канал, которому пользователи доверяют.

Если код проверки совпадает, вероятность MITM-подмены публичных ключей крайне мала.

---

## Что защищает программа

Программа защищает содержимое сообщений при условии, что:

1. устройства пользователей не заражены;
2. приватные ключи не раскрыты;
3. пользователи действительно сравнили код проверки;
4. используется оригинальная версия программы;
5. зашифрованные сообщения не расшифровываются на скомпрометированном устройстве.

AES-GCM также защищает сообщение от незаметного изменения.

Если зашифрованный токен будет изменён, расшифровка завершится ошибкой.

---

## Что программа не защищает

Программа не защищает от:

- заражённого компьютера;
- кейлоггеров;
- вредоносных программ;
- скриншотов экрана;
- утечки через буфер обмена;
- подменённого `.exe`;
- пользователя, который не сверил код проверки;
- пользователя, который сверил код проверки через недоверенный канал;
- компрометации GitHub-аккаунта или ключей подписи;
- ошибок пользователя при копировании и передаче сообщений.

## Как пользоваться

### 1. Запустить два экземпляра программы

Для теста на одном компьютере можно открыть два окна программы.

Например:

```bash
py aes_ecdh_gui_v2.py
```

И во втором терминале ещё раз:

```bash
py aes_ecdh_gui_v2.py
```

---

### 2. Выбрать роли

В первом окне выбрать:

```text
Alice
```

Во втором окне выбрать:

```text
Bob
```

Важно: один пользователь должен быть Alice, второй — Bob.

Если оба выберут одну и ту же роль, обмен будет некорректным.

---

### 3. Сгенерировать ключи

В обоих окнах нажать:

```text
Сгенерировать мой ключ
```

После этого у каждого пользователя появится свой public key.

---

### 4. Обменяться публичными ключами

Alice копирует свой public key и передаёт Bob.

Bob копирует свой public key и передаёт Alice.

Public key не является секретом. Его можно передавать открыто.

---

### 5. Вставить public key собеседника

Alice вставляет public key Bob.

Bob вставляет public key Alice.

Для этого используется поле:

```text
Public key собеседника
```

---

### 6. Выработать AES-ключи

В обоих окнах нажать:

```text
Выработать ключи AES
```

После этого появится код проверки.

---

### 7. Сравнить код проверки

Alice и Bob должны сравнить код проверки.

Если код совпадает, можно шифровать сообщения.

Если код не совпадает, использовать соединение нельзя.

---

### 8. Зашифровать сообщение

Отправитель выбирает режим:

```text
Encrypt
```

Вводит текст сообщения и нажимает:

```text
Зашифровать
```

На выходе получится токен вида:

```text
AECDH1....
```

Этот токен можно отправить собеседнику.

---

### 9. Расшифровать сообщение

Получатель выбирает режим:

```text
Decrypt
```

Вставляет токен `AECDH1...` и нажимает:

```text
Расшифровать
```

Если ключи правильные и сообщение не повреждено, появится исходный текст.

---

## Формат зашифрованного сообщения

Зашифрованное сообщение имеет формат:

```text
AECDH1.base64url(...)
```

Внутри токена находятся:

```text
MAGIC/version
direction
nonce
ciphertext
authentication tag
```

Где `direction` указывает направление сообщения:

```text
Alice -> Bob
Bob -> Alice
```

Для каждого направления используется отдельный AES-ключ.

---

## Почему используется AES-GCM

AES-GCM обеспечивает одновременно:

- шифрование;
- проверку целостности;
- проверку подлинности данных в рамках общего ключа.

Если злоумышленник изменит хотя бы один байт зашифрованного сообщения, расшифровка завершится ошибкой.

---

## Почему важен nonce

AES-GCM требует, чтобы один и тот же nonce не повторялся с одним и тем же ключом.

В программе nonce генерируется через:

```python
os.urandom(12)
```

Размер nonce:

```text
12 bytes / 96 bits
```

Это стандартный размер nonce для AES-GCM.

После сборки можно посчитать SHA-256 хеш:

```powershell
Get-FileHash .\dist\AES_ECDH_Tool.exe -Algorithm SHA256
```

Сохранить хеш в файл:

```powershell
Get-FileHash .\dist\AES_ECDH_Tool.exe -Algorithm SHA256 | Out-File .\dist\AES_ECDH_Tool.exe.sha256.txt
```

Для релиза рекомендуется публиковать SHA-256 рядом с `.exe`.

## GPG-подпись релиза

Подписать `.exe` отдельной GPG-подписью:

```bash
gpg --armor --detach-sign dist/AES_ECDH_Tool.exe
```

После этого появится файл:

```text
dist/AES_ECDH_Tool.exe.asc
```

Для GitHub Release рекомендуется публиковать:

```text
AES_ECDH_Tool.exe
AES_ECDH_Tool.exe.asc
AES_ECDH_Tool.exe.sha256.txt
```

---

## Проверка GPG-подписи

Пользователь может проверить подпись так:

```bash
gpg --verify AES_ECDH_Tool.exe.asc AES_ECDH_Tool.exe
```

Если подпись корректная, GPG покажет сообщение вида:

```text
Good signature
```

Важно: GPG-подпись подтверждает, что файл был подписан владельцем GPG-ключа и не был изменён после подписи.

GPG-подпись не доказывает, что в программе нет ошибок.
